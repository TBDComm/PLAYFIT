## PlayFit 운용 워크플로우

### 핵심 문제 의식

Claude Code는 컨텍스트 창(~200k 토큰) 안에서 동작하고, 세션이 끊기면 기억이 리셋된다. 그래서 두 가지를 최적화해야 함:

1. 연속성 — 세션 간 상태를 어떻게 이어받는가
2. 토큰 효율 — 매 세션 읽어야 할 파일을 얼마나 짧게 유지하는가


### 파일 계층 구조

항상 로드됨 (자동)
├── CLAUDE.md          ← 규칙, 코딩 가이드라인 (최대한 짧게 유지)
└── memory/MEMORY.md   ← 세션 간 기억 인덱스 (200줄 제한)

세션 시작 시 읽음 (필수 1회)
└── HANDOVER.md        ← 현재 상태 + In-Progress Lock + 활성 스텝 스펙

필요할 때만 읽음 (온디맨드)
├── SPEC.md            ← 다음 스텝 시작 시 해당 섹션만
├── SPEC_archive.md    ← 기구현 기능 수정할 때만
├── HANDOVER-archive.md← 과거 결정 디버깅할 때만
└── rules/*.md         ← 해당 영역 코드 작성 전에만


### HANDOVER.md 의 역할 (핵심)

In-Progress Lock     ← 작업 중 채워짐 → 중단해도 안전하게 재개 가능
Current Status       ← 어느 스텝까지 완료됐는지
Active Step Spec     ← SPEC.md 해당 섹션을 여기에 복사
                       → 다음 세션에서 SPEC.md 읽을 필요 없음
Change Log           ← 완료 항목은 archive로 이동 → HANDOVER 계속 짧아짐


### 토큰 절약 메커니즘

기법                          효과
--------------------------------------------------------------
완료 로그 → archive 이동      HANDOVER 크기 고정
스텝 스펙을 HANDOVER에 인라인 SPEC.md 읽기 생략
MEMORY.md는 인덱스만          200줄 제한 내 유지
온디맨드 파일들 분리           필요 없으면 컨텍스트에 안 들어옴


### 세션 흐름

세션 시작
  → HANDOVER.md 읽기
  → Lock 확인 → 있으면 거기서 재개
  → 없으면 Current Status 확인 → 다음 스텝 파악

작업 중
  → Lock 채우기 → 코드 작성 → rules 파일 대조
  → "Stopped at" 계속 업데이트

세션 종료
  → Lock 클리어 → HANDOVER.md 업데이트 → 완료 보고


요약: HANDOVER.md = 세션 간 브레인, MEMORY.md = 장기 기억 인덱스,
나머지는 필요할 때만 꺼내는 구조.
