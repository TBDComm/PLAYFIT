# GUILDELINE — Claude Code 지침 (한국어 참고용)

> 이 파일은 CLAUDE.md의 한국어 번역본입니다. Claude Code는 읽지 않습니다 — 사람이 참고하는 용도입니다.
> 원본: `CLAUDE.md`

---

> ## 필수 — 매 세션, 매 작업, 예외 없음
>
> **작업 전:**
> 1. `HANDOVER.md` 전체 읽기 — 세션 시작 시 유일한 필수 읽기 파일
> 2. **In-Progress Lock** 확인 — 채워져 있으면 이전 세션이 중단된 것 → 거기서 재개
> 3. **Current Status** 확인 — 현재 활성 스텝과 블로커 확인
> 4. 스텝 구현 전 → `SPEC.md`의 해당 섹션만 읽기
>    (HANDOVER.md ACTIVE STEP에 스펙이 인라인으로 있으면 SPEC.md 생략)
>
> **작업 중:**
> - 작업 시작 즉시 In-Progress Lock 채우기
> - "Stopped at" 지속적으로 업데이트 — 언제 중단돼도 안전하게 재개 가능해야 함
>
> **작업 후** (스텝이든 아니든, 아무리 작은 작업이든):
> - Lock 해제, 보고 전에 `HANDOVER.md` 업데이트 — 절대 생략 금지

## 언어 규칙 — 모든 파일

**모든 `.md` 파일의 기본 언어는 영어.** 한국어는 진짜 필요한 경우에만 허용 — 예: 한국어 UI 레이블, 한국 특유의 문화적 맥락, 번역 시 의미가 사라지는 용어. 습관적으로 한국어를 쓰지 말 것. 매 세션 이 규칙이 위반됨 — 작성 전 확인할 것.

## 코딩 규칙

- 주석은 영어
- TypeScript strict mode 전체 적용
- 모든 API 호출(Steam, Claude, Supabase)은 Next.js API routes 경유만 — 프론트엔드에 키 노출 금지
- Claude API 응답은 항상 try-catch + JSON.parse 방어 처리
- 동작하는 코드 먼저, 최적화는 나중에
- 파일 구조는 플랫하게 유지 — 불필요한 중첩 금지

## 능동적 개선 제안 규칙

구현 중 명세보다 명백히 나은 접근법을 발견했을 때 — 문서화된 API 동작, 알려진 기술 제약, 확립된 베스트 프랙티스 기반 — 진행 전에 멈추고 플래그를 세운다.

형식:

> **더 나은 접근법:** [현재 스펙이 말하는 것]
> → [더 나은 접근법]
> → [왜 나은지 — 구체적인 기술적 이유 명시, 의견 아님]
> → 원래 스펙대로 진행할까요, 이 방식으로 전환할까요?

규칙:

- 검증 가능한 사실 기반일 때만 제안 — 추정이나 선호 아님
- 물어보지 않고 다른 방식으로 조용히 구현하지 말 것
- 사소한 스타일 차이는 제안하지 말 것 — 성능, 비용, 신뢰성, 정확성에 의미 있는 영향이 있을 때만
- 한 번에 개선 제안 하나 — 여러 제안 묶기 금지
- 원래대로 하라고 하면 같은 포인트 다시 제기하지 말 것

## 문서 최적화 목표

이 프로젝트의 `.md` 파일을 작성하거나 수정할 때마다 세 가지 목표를 적용한다:

1. **토큰 낭비 최소화** — 다른 곳에 이미 있는 정보는 절대 반복하지 말 것; 현재 작업에 불필요한 내용은 아카이브로
2. **토큰 효율 극대화** — 세션 시작 시 필수 읽기는 최대한 짧게; 세부 내용은 온디맨드 파일(SPEC.md, 아카이브)에
3. **컨텍스트 저하 감소** — 컨텍스트가 커질수록 Claude Code 성능이 떨어진다; 항상 읽히는 파일의 불필요한 줄 하나하나가 매 세션 치르는 비용임

완전성과 충돌할 때: 간결하고 정확한 것을 선택한다. 한 번만 또는 드물게 필요한 정보는 아카이브나 참조 파일에 — HANDOVER.md나 CLAUDE.md에 넣지 말 것.
